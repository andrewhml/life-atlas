## Interpretation guide

- **Bucket 1 population** scopes the byte-degradation problem. If large, iPhone/iCloud re-import is required for those DNGs. If small (< ~500), targeted per-asset re-import is easier than building the full Phase D tool.
- **Bucket 2 population** scopes the metadata-degradation problem. Large → re-import via `immich-go` with Google Takeout mode (which reads JSON sidecars). Small → Phase D can patch individually.
- **Bucket 3 spikes** identify import events. A single-day spike of N thousand correlates 1:1 with Bucket 2 if that import lost metadata. This confirms the cause.
